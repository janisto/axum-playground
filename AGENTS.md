- Correctness first: prioritize correctness, then maintainability, then performance.
- Reflect before acting: after tool results, briefly summarize the local conclusion and choose the best next step.

## Project Overview

Axum Playground is a Rust 2024 application built with Axum 0.8.x and Tokio. It provides a production-oriented Axum API with Rust-native patterns for routing, middleware, authentication, observability, and deployment.

### Key Features

- root health endpoint at `/health`
- versioned API under `/v1`
- OpenAPI document at `/v1/openapi`
- Swagger UI at `/api-docs`
- JSON and CBOR success negotiation on versioned endpoints
- RFC 9457-style Problem Details with JSON and CBOR variants
- strict request ID validation with UUID fallback
- tracing-based request logging with `traceparent` correlation
- Firebase-authenticated profile CRUD with production JWKS verification and emulator support
- Firestore-backed profile persistence with audit logging
- GitHub upstream proxy endpoints with explicit error mapping

### Current Stack

- Rust 1.95, edition 2024
- Axum 0.8.9
- Tokio 1.52.3
- Tower 0.5.x and tower-http 0.7.x
- Utoipa 5.x with utoipa-swagger-ui 9.x
- Reqwest 0.13.x with rustls
- Firestore 0.49.x
- JsonWebToken 10.4.x

### Current Best-Practice Anchors

These repo conventions intentionally follow current guidance:

- app-wide state is shared through `Router::with_state(...)` and `State`, not through ad hoc globals
- middleware stacking uses `tower::ServiceBuilder` and Axum/Tower layers
- graceful shutdown uses `tokio::select!` over `ctrl_c` and `SIGTERM`
- in-process route tests use `tower::ServiceExt::oneshot(...)`
- keep `main.rs` thin and move application construction into reusable library code

## Justfile

The repository uses `just` as the main command surface. Run `just` to list recipes.

Key recipes:

- `just build` - build the application
- `just run` - run the local server
- `just fmt` - apply formatting
- `just fmt-check` - verify formatting
- `just lint` - run clippy with warnings denied
- `just test` - run the main suite with `cargo nextest`
- `just test-doc` - run doctests
- `just test-emulators` - run Firestore emulator tests when the emulator is configured
- `just check` - local verification bundle
- `just ci` - CI-oriented QA bundle
- `just coverage-lcov` - generate `coverage.lcov`
- `just coverage-html` - generate HTML coverage output under `target/llvm-cov/html`
- `just deny` - run dependency policy checks
- `just audit` - run dependency vulnerability checks
- `just docker-build` - build the image, preferring Podman and falling back to Docker

Use `just` for standard repo workflows. For focused validation when there is no precise Justfile target, targeted `cargo test --locked --test ...` commands are acceptable only when the dependency graph is expected to stay unchanged.

### Cargo.lock

- `Cargo.lock` is generated and maintained by Cargo. Do not edit it by hand.
- This repository is an application, so keep `Cargo.lock` checked in.
- Use `--locked` only for deterministic validation when the existing lockfile should remain unchanged. Cargo will error if the lockfile is missing or if resolution would change.
- If a manifest or feature change requires a lockfile update, let Cargo rewrite `Cargo.lock` by running the relevant Cargo command without `--locked`, then rerun the locked validation step.
- Treat `cargo update` and `cargo generate-lockfile` as intentional lockfile-refresh commands. They can rewrite dependency resolution broadly, so do not use them as a routine fix for a small manifest edit.
- If a lockfile-refresh command introduces unrelated churn, stop and ask before proceeding instead of compensating with manual `Cargo.lock` edits.

## Architecture Principles

### Application Construction

- Keep `main.rs` thin: config loading, tracing init, state creation, listener binding, and `axum::serve(...)` only.
- Put reusable router construction in `lib.rs` and `app.rs`.
- Prefer `build_app(...)` and `build_app_with_routes(...)` for tests and composition rather than rebuilding router stacks ad hoc.

### State

- App-wide state lives in `AppState` and is shared through `Arc<AppState>`.
- Route modules should use `State<Arc<AppState>>` only when they need services or config.
- Keep handlers stateless when state is unnecessary.

### Router Layout

- Keep `/health` at the root.
- Keep Swagger UI at `/api-docs`.
- Keep versioned API routes and the OpenAPI document under `/v1`.
- Add new route modules under `src/http/v1/` unless the route is intentionally root-level.

### Middleware

- Middleware order is defined centrally in `src/app.rs`; do not scatter cross-cutting behavior across handlers.
- Prefer `tower::ServiceBuilder` composition for multiple layers.
- Keep request ID, panic recovery, logging, security headers, CORS, and timeout policy in middleware.
- If a route needs a middleware exception, keep the exception narrow and explicit.

### Shutdown & Runtime

- Use `tokio::select!` for graceful shutdown coordination.
- Preserve the current `ctrl_c` plus Unix `SIGTERM` behavior in `src/shutdown.rs`.
- Keep Cloud Run compatibility: `0.0.0.0:$PORT`, no TLS termination in-process, non-persistent filesystem assumptions.

## Coding Conventions

### Handler Pattern

- Expose `pub fn router() -> Router<Arc<AppState>>` from each route module.
- Prefer small handlers with explicit inputs and one clear response path.
- Use shared response helpers from `src/http/codec.rs` rather than hand-rolling JSON or CBOR responses.

Common helpers:

- `success_response(...)`
- `success_response_with_headers(...)`
- `decode_request_body(...)`
- `problem_response(...)`

### OpenAPI

- Document public handlers with `#[utoipa::path(...)]`.
- Keep documented paths aligned with actual external routes.
- When adding versioned endpoints, include both JSON and CBOR media types when the runtime supports both.

### Error Handling

- Use Problem Details consistently for non-success paths.
- Keep error messages stable and client-safe.
- Return explicit status mappings for validation, cursor, auth, conflict, and upstream failures.
- Do not leak internal error strings, stack traces, credentials, or upstream raw responses to clients.

### Request Decoding & Validation

- For JSON/CBOR request bodies, use `decode_request_body(...)`.
- Validate inputs explicitly with `validator` or route-specific checks after decoding.
- Use `400 Bad Request` for malformed syntax or cursor format failures.
- Use `422 Unprocessable Entity` for validation failures on otherwise well-formed input.

### Serialization & Naming

- Use camelCase JSON property names.
- Use `#[serde(rename = "...")]` when Rust field naming differs from the API contract.
- Keep storage-specific naming separate from API naming. Firestore document shapes can diverge when needed.

### Auth & Services

- Route-level auth behavior should stay consistent with `src/auth/mod.rs`.
- Preserve the production path: Google JWKS verification plus Identity Platform lookup for disabled and revoked users.
- Preserve emulator behavior when `FIREBASE_AUTH_EMULATOR_HOST` is configured.
- Use mock services for fast in-process tests and Firestore-backed services only when the code path requires them.

### Dependencies

- Avoid new dependencies unless the standard library or existing crates clearly cannot solve the problem.
- Prefer crates with active maintenance, current docs, and broad adoption.
- Keep the dependency surface justified by actual feature needs.

## REST API Guidelines

### URI Design

- Keep versioned API paths under `/v1`.
- Use plural nouns for collections.
- Avoid verbs in URIs when the HTTP method already expresses the action.
- Keep nesting shallow and purposeful.

### HTTP Methods & Status Codes

Use these defaults unless the existing contract requires otherwise:

| Method | Purpose | Typical Success Status |
| --- | --- | --- |
| GET | Retrieve resource(s) | 200 OK |
| POST | Create a resource | 201 Created |
| PATCH | Partial update | 200 OK |
| DELETE | Remove a resource | 204 No Content |

Common error statuses:

| Status | Use Case |
| --- | --- |
| 400 | malformed syntax, invalid cursor, cursor type mismatch |
| 401 | missing or invalid authentication |
| 403 | authenticated but not authorized |
| 404 | resource not found |
| 405 | method not allowed |
| 409 | conflict or duplicate write |
| 422 | validation failure |
| 500 | unexpected internal failure |
| 502 | upstream dependency failure |
| 503 | authentication or certificate dependency unavailable |

### Content Types

Requests:

- `application/json`
- `application/cbor`

Responses:

- success responses on versioned endpoints negotiate between `application/json` and `application/cbor`
- `/health` stays JSON only
- errors use `application/problem+json` or `application/problem+cbor`

### Request ID

- Respect valid incoming `X-Request-Id` values.
- Generate UUIDv4 when the incoming value is missing or invalid.
- Keep the strict validation rules already implemented.

### Pagination

- Use cursor-based pagination through `src/pagination`.
- Invalid cursors must return `400 Bad Request`.
- Preserve RFC 8288 `Link` header behavior.
- Reuse `paginate(...)` and the existing cursor helper types instead of open-coding pagination.

## Testing Guidelines

### Test Structure

- Prefer integration tests under `tests/` for HTTP behavior.
- Keep pure helper logic unit-tested close to the module when appropriate.
- Reuse `tests/common/mod.rs` helpers instead of duplicating response decoding or state setup.

### Preferred Test Pattern

- Build the app in-process with `build_app(...)`.
- Send requests through `tower::ServiceExt::oneshot(...)`.
- Assert both status codes and contract-relevant headers.
- For JSON/CBOR behavior, decode the body with the matching shared helper.

### What to Test

- success paths
- validation failures
- auth failures
- pagination headers and cursor handling
- Problem Details structure
- OpenAPI path coverage for new public routes when relevant

### Emulator Tests

- Firestore emulator coverage lives in `tests/firestore_emulator.rs`.
- Keep emulator-backed tests conditional and isolated.
- Use the mock profile and auth services for the default fast path when emulator behavior is not under test.

### Validation Commands

Use the narrowest relevant command first when the lockfile is expected to remain unchanged:

- `cargo test --locked --test api_health`
- `cargo test --locked --test api_hello`
- `cargo test --locked --test api_items`
- `cargo test --locked --test api_profile`
- `cargo test --locked --test firestore_emulator`

If `Cargo.toml` or dependency features changed and Cargo needs to refresh `Cargo.lock`, run the narrow validation command once without `--locked`, then rerun the locked command after the lockfile settles.

Then use repo-level checks as appropriate:

- `just fmt-check`
- `just lint`
- `just test`
- `just test-doc`
- `just test-emulators`

For larger change sets, finish with `just check` or `just ci` when practical.
